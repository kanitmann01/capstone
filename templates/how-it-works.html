Behind the check

How Brand Guard scores a link

When you paste a URL, the app fetches what it can from the live page, runs several independent checks, then blends those signals into one risk score and a simple verdict. This page is a plain-language map of that process.

What you should expect

Brand Guard is built for quick triage, not courtroom evidence. A high score means several automated signals agreed the page looks risky; a low score means they did not. Network errors, blocked pages, or missing content can limit what the tools see, which is why the interface also lists which checks ran and which could not.

End-to-end flow

From the URL you submit to the verdict on screen
  1. 1

    Normalize the address

    The URL is cleaned and canonicalised so host names, schemes, and obvious typos are handled consistently before anything else runs.

  2. 2

    Build a page snapshot

    The server tries to load the page like a browser would and capture HTML, visible text, forms, hosting hints, and related metadata. That snapshot feeds most of the downstream checks.

  3. 3

    Run signals in parallel

    Several subsystems read the snapshot or the hostname at the same time: a rules-based scorer, a FastText model trained on URL and page text, URL heuristics, TLS certificate checks, domain-age signals, optional threat-intel lookups, a structured feature model, and a brand-lookalike detector that works from the URL alone.

  4. 4

    Blend into one score

    Each family of checks produces a risk contribution where available. Those contributions are combined with fixed weights, then compared to a configured threshold to label the page as phishing-like or clean for this demo.

  5. 5

    Store a record (when enabled)

    Successful scans can be written to the local dataset store so researchers can audit examples and retrain models from the Data & evaluation page.

How much each signal family weighs

Weights apply only to components that returned a score for that request. If a model is offline or a check fails, its slice is omitted and the remaining weights are renormalized so the verdict still reflects what actually ran.

Rules baseline
30% Interpretable rules over the snapshot (for example login forms, suspicious phrases, host mismatches).
FastText
30% Text classifier over the captured page and URL context.
Legacy bundle
15% Combined heuristics, SSL, domain age, and threat intel when those probes succeed.
Brand recognition
15% Domain-only analysis for typosquats and impersonation patterns.
Structured ML
10% A second model over engineered features derived from the URL and legacy check outputs.

If brand recognition reports a high-confidence impersonation scam, the composite score is not allowed to sit below that brand score—so obvious spoofs are not averaged away.

Dataset & methodology (v3)

The current models are trained on a balanced, host-disjoint dataset built from fresh phishing feeds and legitimate bank domains.

Phishing sources

  • OpenPhish public feed (300 URLs)
  • URLhaus parsed live feed (500 URLs)
  • Host-disjoint split: no domain appears in both train and test

Legitimate sources

  • Tranco top-list domains filtered to bank/brand/SaaS
  • Direct official domains for 100 global banks
  • Balanced 50/50 with phishing class
1,600 total URLs 800 phishing · 800 clean 70/15/15 train/val/test split 110 brands in inventory
